Game 2 — What I Learned Building "Hoop Hustle"

March 2026 · From static clicks to real-time physics
Play the Game

The Idea

Basketballs fall from the sky. You move a hoop left and right to catch them. Bombs fall too — catch one and you lose a life. Survive 60 seconds and you advance to the next round, where everything gets faster.

Simple on paper. But building this meant jumping from static button clicks to real-time physics — gravity, velocity, collision detection, and a game loop running 60 times per second. Game 1 was a conversation. This game is a simulation.

Arcade Physics — Things That Fall

Game 1 had no physics engine. Nothing moved on its own — the player clicked, and the game responded. In Hoop Hustle, objects fall automatically. Phaser's Arcade Physics handles this with a single config: 

physics: {
    default: 'arcade',
    arcade: {
        gravity: { y: 300 },  // Pull everything downward
        debug: false
    }
}

The key insight: gravity is acceleration, not speed. At y: 300, objects gain 300 pixels/second every second. They start slow and get faster — that natural "falling" feel comes from the math, not from manual animation.

But not everything should fall. The basket stays at the bottom. The score text floats in place. You choose which objects participate in physics: 

// This object falls (gravity ON by default)
this.physics.add.existing(ball);

// This object stays put
basket.body.setAllowGravity(false);

Takeaway: A physics engine isn't all-or-nothing. You decide which objects live in the physics world and which are just pictures on screen.

Physics Groups — Managing the Swarm

When you spawn a new basketball every 1.5 seconds, you quickly have dozens of them falling at once. Checking each one individually against the basket would be tedious. Phaser's Physics Groups solve this: 

// Create a group for all basketballs
this.balls = this.physics.add.group();

// Spawn a ball into the group
const ball = this.add.sprite(x, 0, 'basketball');
this.balls.add(ball);

// Check ALL balls against the basket in one line
this.physics.add.overlap(
    this.basket, this.balls, this.catchBall, null, this
);

I kept balls and bombs in separate groups. They look similar — both are falling circles — but they trigger completely different logic. Catching a ball means +10 points. Catching a bomb means losing a life. One overlap rule per group keeps the code clean: 

// Two groups, two rules, two callbacks
this.physics.add.overlap(basket, this.balls, this.catchBall, null, this);
this.physics.add.overlap(basket, this.bombs, this.hitBomb, null, this);

The Invisible Problem — Memory Leaks

This was the most eye-opening lesson. Once a ball falls past the bottom of the screen, the player can't see it. But the computer still tracks it. The physics engine still calculates its position, velocity, and potential collisions — every single frame.

After 2 minutes of gameplay, there could be 80+ invisible objects being processed 60 times per second. In a simple game, that's fine. In a bullet-hell with thousands of projectiles? Lag, stutter, crash. 

// In update() — runs every frame
this.balls.getChildren().forEach(ball => {
    if (ball.y > 660) {
        ball.destroy();  // Gone from memory
    }
});

Takeaway: What the player can't see, the game shouldn't be tracking. Always clean up objects that leave the screen.

Spawning and Timed Events

Games need things to happen on a schedule. Phaser's time.addEvent() lets you say "do this every N milliseconds, forever": 

// Spawn a ball every 1.5 seconds
this.time.addEvent({
    delay: 1500,
    callback: this.spawnBall,
    callbackScope: this,
    loop: true
});

For bombs, I wanted a 10-second safe period before they start appearing — giving the player time to get comfortable. I used delayedCall to wait, then started the bomb spawner inside the callback: 

// Wait 10 seconds, THEN start spawning bombs
this.time.delayedCall(10000, () => {
    this.time.addEvent({
        delay: 3000,
        callback: this.spawnBomb,
        callbackScope: this,
        loop: true
    });
});

The Round System — A Design Decision

Originally, difficulty increased gradually within a single 60-second round. But I realized a round-based system feels better: survive 60 seconds, see your stats, choose to continue or quit.

The first approach tried to handle round transitions inside GameScene — show a "Round Complete!" message, wait 2 seconds, resume. But the physics engine kept running in the background. Balls were still falling behind the announcement text.

The fix: a dedicated PromoteScene. When a round ends, GameScene fully stops and hands off to PromoteScene. No background physics, no race conditions. The player sees their stats, previews the next round's difficulty, and clicks when they're ready. 

// GameScene → done with round
this.scene.start('PromoteScene', {
    score: this.score,
    lives: this.lives,
    round: this.round
});

// PromoteScene → player clicks "Next Round"
this.scene.start('GameScene', {
    score: score,
    lives: lives,
    round: nextRound   // State carries over
});

Takeaway: Each scene should do one job. If you're fighting the physics engine to show a UI screen, you probably need a separate scene.

Difficulty Scaling

Each round gets harder in three ways:

  • Gravity increases — +150 per round (300 → 450 → 600...)
  • Starting velocity increases — balls already moving when they spawn
  • Bombs appear sooner — 10s delay → 5s → immediate

I initially only increased gravity, but the difference was subtle — gravity is acceleration, so the speed difference only shows late in the fall. Adding an initial velocity that scales per round made the difficulty jump immediately noticeable: 

ball.body.setVelocityY(50 + (this.round - 1) * 80);
// Round 1: 50 px/s  → gentle start
// Round 2: 130 px/s → noticeably faster
// Round 3: 210 px/s → fast right away

The Architecture Refactor

By the time I had gameplay, visuals, UI, textures, and round logic working, GameScene was ~280 lines doing too many jobs. I broke it apart: 

02-catch-and-collect/
  assets/
    textures.js      ← all texture generation
    background.js    ← gradient sky, clouds
  objects/
    Basketball.js    ← ball creation + physics
    Bomb.js          ← bomb creation + physics
    Basket.js        ← player movement + physics
  utils/
    ButtonHelper.js  ← reusable styled buttons
    UIHelper.js      ← HUD, titles, labels
  scenes/
    MenuScene.js     ← start screen
    GameScene.js     ← gameplay only (~120 lines)
    PromoteScene.js  ← round transition
    GameOverScene.js ← final results

After the refactor, GameScene dropped to ~120 lines of pure game logic. Each file has one responsibility. Adding sounds or animations later means extending the object classes — not bloating the main scene.

Takeaway: Refactor when the code starts fighting you. If you're scrolling past texture-drawing code to find your game logic, it's time to separate concerns.

Mistakes I Made

  1. Hardcoded screen boundaries — I wrote manual if checks to keep the basket on screen, but the numbers were tied to the basket's width. Changing the width broke the boundaries silently. Fix: setCollideWorldBounds(true) — one line, handles any size.
  2. Wrong text property — Used fontWeight: 'bold' instead of fontStyle: 'bold'. Phaser silently ignored it. No error, no warning, just text that wasn't bold. Silent failures are the hardest bugs.
  3. Double game-over trigger — Timer hitting zero and last life lost could fire in the same frame. Fix: a simple isGameOver boolean checked at the top of every method that could end the game.
  4. Memory leak from off-screen objects — Balls fell past the screen and lived forever in memory. Fix: destroy anything past y > 660 in the update loop.

What I'd Do Differently

I'd create a global theme config from the start. I kept typing hex codes like 0xFF8C00 and 0xFFA500 throughout the codebase. A single theme.js file with named colors — BASKETBALL_ORANGE, BOMB_DARK, SKY_BLUE — would make the code more readable and make it trivial to reskin the game later.

I'd also use the framework's built-in features earlier. Every time I wrote custom logic, Phaser already had a cleaner one-liner. Check the docs first, code second.

What's Next

The foundation is solid and the architecture is ready to scale. Next steps: sound effects (catch swoosh, bomb explosion, background music), animations (ball spin, screen shake on bomb hit), and mobile touch controls. Time to make it feel like a real game.
