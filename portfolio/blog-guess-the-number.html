Game 1 — What I Learned Building "Guess the Number"

March 2026 · First game in my game dev journey
Play the Game

The Idea

A simple concept: the computer picks a secret number, you try to guess it. But even a simple idea can teach a lot when you build it as a real game — with menus, difficulty levels, a timer mode, score tracking, and visual feedback.

I built this using Phaser.js 3, a 2D game framework for the browser. The goal wasn't just to make the game work — it was to learn how games are structured under the hood.

Why a Game Framework? Why Not Just HTML?

My first instinct was "just use HTML buttons and divs." But games update 60 times per second. Every time you change an HTML element, the browser recalculates layout — that's called reflow. Fine for websites, terrible for games.

A <canvas> element solves this. It's like a drawing board — you draw pixels directly, no layout recalculation. Phaser.js handles the canvas for you and gives you scenes, input handling, timers, and a game loop out of the box.

Takeaway: HTML is for documents. Canvas is for real-time rendering. A framework like Phaser bridges the gap so you don't have to manage the canvas manually.

Game Config — The Entry Point

Every Phaser game starts with a config object. This is where you define the canvas size, the renderer, and which scenes to load: 

const config = {
    type: Phaser.AUTO,       // WebGL or Canvas — Phaser picks
    width: 800, height: 600, // Game resolution
    scene: [MenuScene, GameScene, GameOverScene],
    scale: {
        mode: Phaser.Scale.FIT,
        autoCenter: Phaser.Scale.CENTER_BOTH
    }
};

One decision I made early: no physics engine. Phaser supports Arcade Physics and Matter.js, but this game has no moving objects or gravity. Don't add what you don't need.

Scenes — The Building Blocks

Scenes are like screens. Each one has its own create() method that runs when the scene starts. My game has three:

  • MenuScene — Title, difficulty selection, mode selection
  • GameScene — The actual guessing gameplay
  • GameOverScene — Results, best score, play again

Switching between scenes is straightforward:

// Start a new scene and pass data to it
this.scene.start('GameScene', {
    difficulty: 'Easy',
    maxNumber: 50,
    mode: 'relax'
});

// Receive that data in the next scene
create(data) {
    console.log(data.difficulty); // "Easy"
}

Bug I hit: I renamed a property from range to max in one place but forgot to update every scene that reads it. Lesson: when you change a data structure, search for all usages.

Making Things Clickable

Phaser has no built-in buttons. Instead, you make any object interactive: 

text.setInteractive({ useHandCursor: true })
    .on('pointerover', () => text.setTint(0x00ff88))
    .on('pointerout', () => text.clearTint())
    .on('pointerdown', () => { /* click action */ });

For the difficulty buttons, I used a data-driven approach — an array of objects with a forEach loop. Adding a 4th difficulty would mean adding one object to the array. No copy-pasting button code.

Two Game Modes

I added two ways to play:

  • Relax mode — No time limit, but only 10 attempts
  • Timer mode — Race the clock (30s / 60s / 90s depending on difficulty), unlimited attempts

Each mode has exactly one pressure mechanic. Relax limits your guesses. Timer limits your time. This keeps each mode feeling distinct.

The timer runs on a Phaser timed event that fires every second. When it hits 5 seconds remaining, the display turns red — a small touch that adds urgency.

Visibility vs Interactivity

One sneaky bug pattern I learned about: ghost clicks. If you hide a button with setVisible(false) but forget to call disableInteractive(), the invisible button is still clickable. Always pair them: 

// Hide properly
btn.setVisible(false).disableInteractive();

// Show properly
btn.setVisible(true).setInteractive();

Saving Best Scores

I used the browser's localStorage to persist best scores across sessions. Each difficulty + mode combo gets its own key: best_Easy_relax, best_Hard_timer, etc.

The save logic lives in GameScene, which checks if the current attempt count beats the stored best. It passes a simple isNewBest flag to GameOverScene — instead of having GameOverScene re-compute the comparison. Don't duplicate logic across scenes.

Cleanup Matters

I used a hybrid approach: Phaser canvas for the game, but a real HTML <input> for typing guesses (better UX). The catch? When switching scenes, Phaser clears the canvas but doesn't touch HTML elements.

Without cleanup, the input field stays visible on the Game Over screen. The fix is registering a shutdown handler: 

this.events.on('shutdown', this.shutdown, this);

Takeaway: Phaser manages Phaser objects. Anything outside its world (HTML elements, event listeners, timers) is your responsibility to clean up.

Mistakes I Made

  1. Renamed a data property but didn't update all references — caused silent bugs where values were undefined
  2. Mixed CSS flexbox with Phaser's scale centering — they fought each other. Let one system handle positioning.
  3. Added a physics config I didn't need — removed it. Start lean, add when necessary.
  4. Duplicated logic between scenes — computed best score comparison in two places, causing a bug where this.attempts didn't exist in GameOverScene

What I'd Do Differently

If I rebuilt this game from scratch, I'd plan the data flow between scenes on paper first. Most of my bugs came from passing data between scenes — renaming properties, accessing variables that don't exist in the receiving scene, duplicating logic. A quick sketch of "what data goes where" would have saved debugging time.
